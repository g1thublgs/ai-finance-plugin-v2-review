# 会议费审核第一轮开发报告

## 一、开发范围

第一轮会议费场景以 `00_共用规范文档` 为最高优先级，会议费业务逻辑限定在以下目录：

- `新插件源码/backend/server/src/domain/scenarios/meeting/`
- `新插件源码/backend/server/src/services/rules/python/meeting/`

第一轮同时做了少量公共前端接入适配：

- `新插件源码/预填预审事中审核一体化插件/content.js`
- `新插件源码/预填预审事中审核一体化插件/popup.js`

公共前端修改原因：会议费页面字段需要从财务页面提取并传入后端预填归集；会议费上传和页面审核需要调用既有 `/api/plugin/prefill`、`/api/plugin/audit`；会议费住宿清单不应在会议场景下被当作差旅住宿清单排除。

影响范围：仅新增或补齐会议费分支、会议费字段映射和会议场景下 OCR 类型展示判断。未修改培训费、公务接待费、差旅费、其他事项核心业务目录，未恢复或复制差旅费核心目录。回退方式：撤回 `content.js` 的会议字段提取补充项，以及 `popup.js` 的会议费分支和会议场景 OCR 类型判断即可。

## 三、OCR 和字段模型

`ocrPrompt.js` 与 `ocrProfile.js` 当前支持 9 类会议费材料：

- `meetingNotice`
- `meetingApproval`
- `meetingPlan`
- `attendanceList`
- `meetingSettlement`
- `accommodationList`
- `normalInvoice`
- `paymentProof`
- `other`

代码中未发现“不要返回 accommodationList”之类与会议费规则冲突的表述。OCR prompt 明确禁止输出差旅费、培训费、公务接待费专用类型。

`formSchema.js` 已包含第一轮规则判断所需的基础信息、会议信息、材料完整性字段、费用字段和附件张数字段，包括 `meetingCategory`、`meetingPlanNo`、`approvalNo`、`hasInvoice`、`hasPaymentProof`、`attachmentCount` 等。

## 四、预填归集

`prefillModel.js` 当前输出：

- `ocrItems`
- `records`
- `summary`
- `evidence`
- `warnings`
- `summary.pageFields`

`summary` 已归集会议名称、日期、地点、类别、计划序号、审批号、天数、人数、材料布尔值、住宿费、伙食费、场地租金、资料费、交通费、其他费用、发票金额、支付金额、申请金额、合计金额、附件数等字段。

页面字段映射包括：

- `SQ_MC` -> `title` / `meetingName`
- `SQ_SY` -> `reason`
- `JBR_MC` / `SQ_JBR` -> `applicantName`
- `SSBM_MC` / `SQ_SSBM` -> `departmentName`
- `HYTS` -> `meetingDays`
- `HYRS` -> `attendeeCount`
- `HSF` -> `mealAmount`
- `ZSF` -> `accommodationAmount`
- `CDF` -> `venueRentAmount`
- `QTFY` -> `otherAmount`
- `SQ_JE` -> `applyAmount` / `totalAmount`
- `PJZS` -> `attachmentCount`
- `HYPXBH` -> `meetingPlanNo`
- `HYJYBH` -> `approvalNo`
- `BZ` -> `remark`

归集阶段只整理字段、证据、缺失和冲突提示，不直接写违规结论。页面与 OCR 出现不同值时写入 `warnings` 和 `evidence`，不静默覆盖。
